## 5. 这轮踩过的关键坑

### 5.1 ESM / CJS 边界坑

仓库 `package.json` 里有：

```json
{
  "type": "module"
}
```

但评论脚本最初是 CommonJS 写法，直接运行会报：

- `require is not defined in ES module scope`
- `module is not defined in ES module scope`

当前修复方式：

- 保留兼容入口：`comment_export.js`
- 实际逻辑迁到：`comment_export.cjs`
- 纯逻辑迁到：`lib/comment_export_core.cjs`

所以后续继续改评论抓取，优先改 `.cjs` 文件，不要再改回 `.js + require` 的混合状态。

### 5.2 `4010` 高频风控

这个站会频繁返回 `4010`，处理方式是：

1. 检测到 `4010` 后立刻重新登录
2. 连续命中时切换新的 `devID`
3. 仍然失败时再走延后重试

### 5.3 传输层与登录 HTML 拦截页

在本地代理环境里，出现过两类非业务错误：

1. `curl: (16) Error in the HTTP2 framing layer`
2. 登录接口返回 HTML，而不是 JSON
3. 登录响应解出来后不含 `token`

第二类错误会表现为：

```text
SyntaxError: Unexpected token '<', "<!DOCTYPE "... is not valid JSON
```

这更像 CloudFront 或代理链路返回的拦截页，不是接口协议变了。

第三类错误会表现为：

```text
Error: 登录后未拿到 token
```

这类情况现在也按“登录瞬时异常”处理：

- 登录函数内部会再次重试
- 同时切换新的 `devID`
- 如果最终仍失败，也会按可延后错误处理，而不是直接记成永久失败

### 5.4 “同进程 async 并发”是假的

评论脚本底层请求用的是：

```js
spawnSync('curl', ...)
```

这意味着：

- 同一个 Node 进程里的 Promise 并发不会带来真实网络并发
- 真正要加速，只能用“多进程分片”

当前已经补了稳定分片能力：

- `WORKER_TOTAL`
- `WORKER_INDEX`

按文件列表稳定取模分片，避免两个 worker 重复抓同一个视频。

```bash
cd /opt/zip/jiuyao

  node comment_export.cjs > logs/comment.worker-0.log 2>&1 &
```
